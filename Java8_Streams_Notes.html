01Why Java 8? Motivation & Overview
The Problem Before Java 8

Java is an object-oriented language, but before Java 8, processing a collection required writing verbose for-loops, if-else blocks, and anonymous inner classes. Even a simple task like "find all even numbers" demanded multiple lines of boilerplate — more code means more places to introduce bugs and worse readability.

Key Features Introduced in Java 8
  • Lambda Expressions — anonymous functions; eliminate boilerplate; enable functional style
  • Streams API — declarative, functional data-processing pipeline for collections
  • Date and Time API (java.time) — modern, immutable replacement for java.util.Date
  • Functional Interfaces — interfaces with exactly one abstract method; the foundation for lambdas
  • Method References — shorthand notation for lambdas that simply call an existing method
  • Optional<T> — a container to safely represent nullable values
  • Default & Static methods in interfaces — backward-compatible interface evolution
💡
Core Idea

Java 8 blended OOP with Functional Programming. Functions became first-class values — you can store them in variables, pass them as arguments, and return them from methods. This is what enables the entire Streams API.

⭐
Interview Favourite

Interviewers always ask "What are the main features of Java 8?" Know Lambda, Streams, Functional Interfaces, Method References, Optional, and the new Date/Time API cold.

02Lambda Expressions
What is a Lambda?

A lambda expression is an anonymous function — it has no name, no explicit return type, and no access modifier. It is a compact way to represent a block of code that implements a single-method interface (a functional interface). Lambdas are the syntactic backbone of the entire Streams API.

Syntax Forms
LambdaSyntax.java
// Full form: parameters -> { body }
(int a, int b) -> { return a + b; }

// Single expression: no braces, no return keyword needed
(int a, int b) -> a + b

// No parameters
() -> System.out.println("Hello")

// Single parameter — parentheses are optional
x -> x * 2

// Type inference — compiler infers types from the target interface
(a, b) -> a + b
Before vs After: Runnable Example
RunnableDemo.java
// BEFORE Java 8: Required a separate named class
class Task implements Runnable {
    @Override
    public void run() {
        System.out.println("Hello from task");  // entire class just for this line
    }
}
Runnable r1 = new Task();

// JAVA 8 Lambda: one line replaces the entire class
Runnable r2 = () -> System.out.println("Hello from lambda");
r1.run();  r2.run();
▶ OUTPUT
Hello from task Hello from lambda
Step-by-Step Conversion: Method to Lambda

To convert any method to a lambda: 1 Remove the access modifier. 2 Remove the return type. 3 Remove the method name. 4 Add -> between parameters and body. 5 For single-expression bodies, drop braces and the return keyword.

Conversion.java
// Original method: public int add(int a, int b) { return a + b; }
// Step 1 remove modifier:     int add(int a, int b) { return a + b; }
// Step 2 remove return type:      add(int a, int b) { return a + b; }
// Step 3 remove name:              (int a, int b) { return a + b; }
// Step 4 add arrow:                (int a, int b) -> { return a + b; }
// Step 5 single expression:        (a, b) -> a + b   <-- final lambda
⚠️
Effectively Final Rule

Variables captured from the enclosing scope inside a lambda must be final or effectively final (never reassigned after initialisation). The compiler enforces this because lambdas may execute on a different thread or at a later time.

✅
Best Practice

Keep lambda bodies short — ideally one expression. If a lambda grows beyond 2–3 lines, extract it into a named method and use a method reference. This dramatically improves readability and testability.

03Functional Interfaces
Definition

A functional interface has exactly one abstract method. It may have any number of default and static methods. The @FunctionalInterface annotation is optional but strongly recommended — it causes a compile error if you accidentally add a second abstract method.

FunctionalInterfaceDemo.java
@FunctionalInterface                         // optional but recommended
interface MathOperation {
    int operate(int a, int b);            // exactly ONE abstract method

    default void describe() {             // default method: doesn't count
        System.out.println("I am a math op");
    }
}

// Implement via lambda — no class needed
MathOperation add      = (a, b) -> a + b;
MathOperation subtract = (a, b) -> a - b;
MathOperation multiply = (a, b) -> a * b;

System.out.println(add.operate(5, 3));      // 8
System.out.println(subtract.operate(5, 3)); // 2
System.out.println(multiply.operate(5, 3)); // 15
▶ OUTPUT
8 2 15
💡
Functional Programming in Practice

We are storing a function in a variable: MathOperation add = (a,b) -> a+b. In functional programming, functions are first-class values — they can be stored, passed as parameters, and returned from other functions. This is the foundation of the Streams API.

Java 8 ships with a rich set of built-in functional interfaces in java.util.function. The four most important:

InterfaceMethodTakesReturnsPurpose
Predicate<T>test(T t)TbooleanTest a condition

04Predicate<T> — Test a Condition

Predicate<T> has the method boolean test(T t). You give it an element; it returns true or false. Use it for filtering, validation, and conditional checks. It provides .and(), .or(), and .negate() to compose predicates.

PredicateDemo.java
import java.util.function.Predicate;

// Basic predicate: is the number even?
Predicate<Integer> isEven = x -> x % 2 == 0;
System.out.println(isEven.test(4));             // true
System.out.println(isEven.test(7));             // false

// String predicate
Predicate<String> startsA = x -> x.startsWith("A");
Predicate<String> endsT   = x -> x.endsWith("t");

// AND: both conditions must pass
Predicate<String> both   = startsA.and(endsT);
System.out.println(both.test("Ankit"));          // true (starts A AND ends t)
System.out.println(both.test("Akshay"));         // false (ends y, not t)

// OR: at least one condition passes
Predicate<String> either = startsA.or(endsT);
System.out.println(either.test("Akshay"));        // true (starts with A)

// NEGATE: reverses the result
Predicate<String> notA   = startsA.negate();
System.out.println(notA.test("Rahul"));           // true (does NOT start with A)
▶ OUTPUT
true false true false true true
💡
Composing Predicates

.and(), .or(), .negate() each return a new Predicate — they do not modify the original. You can chain them indefinitely. For case-insensitive checks, normalise first: x -> x.toLowerCase().startsWith("a").

05Function<T,R> — Transform a Value

Function<T,R> has the method R apply(T t). It takes one input of type T and produces a result of type R. Think of it as a conversion or transformation — Integer to String, String to its length, number to its square.

FunctionDemo.java
import java.util.function.Function;

Function<Integer, Integer> doubleIt = x -> 2 * x;
Function<Integer, Integer> tripleIt = x -> 3 * x;

System.out.println(doubleIt.apply(10));              // 20

// andThen: apply doubleIt FIRST, then tripleIt to the result
Function<Integer, Integer> dThenT = doubleIt.andThen(tripleIt);
System.out.println(dThenT.apply(20));               // 20*2=40, 40*3 = 120

// compose: apply tripleIt FIRST, then doubleIt (reverse order)
Function<Integer, Integer> tFirst = doubleIt.compose(tripleIt);
System.out.println(tFirst.apply(10));                // 10*3=30, 30*2 = 60

// identity: returns the input unchanged (useful as a no-op transformer)
Function<Integer, Integer> same = Function.identity();
System.out.println(same.apply(5));                  // 5
▶ OUTPUT
20 120 60 5
💡
andThen vs compose Memory Trick

andThen(g): reads left-to-right — "do this, THEN do g". compose(g): reads right-to-left (math-style) — "do g first, THEN apply this". When in doubt, prefer andThen — it's more intuitive.

06Consumer<T> & Supplier<T>
Consumer<T> — Takes Something, Returns Nothing

Consumer<T> has the method void accept(T t). It consumes a value and does something with it (prints, saves, logs) but returns nothing. Think of it like eating: something goes in, nothing comes out. It provides .andThen() to chain consumers sequentially.

ConsumerDemo.java
import java.util.function.Consumer;

Consumer<Integer> print  = x -> System.out.println(x);
print.accept(25);                                        // 25

// andThen: chain two consumers
Consumer<String> upper = s -> System.out.println(s.toUpperCase());
Consumer<String> lower = s -> System.out.println(s.toLowerCase());
Consumer<String> both  = upper.andThen(lower);              // upper runs first
both.accept("Hello");                                     // HELLO then hello

// Consumer with a list — common pattern with stream.forEach()
List<Integer> nums = Arrays.asList(1, 2, 3);
Consumer<List<Integer>> printAll = list -> {
    for (int n : list) System.out.print(n + " ");
};
printAll.accept(nums);                                    // 1 2 3
▶ OUTPUT
25 HELLO hello 1 2 3
Supplier<T> — Returns Something, Takes Nothing

Supplier<T> has the method T get(). It takes no arguments and returns a value. Use it for lazy initialisation, factory methods, or value generation.

SupplierDemo.java
import java.util.function.Supplier;

Supplier<String> greeting = () -> "Hello, World!";
System.out.println(greeting.get());                     // Hello, World!

// Lazy DB connection — only established when get() is called
// Supplier<Connection> db = () -> DriverManager.getConnection(url);

// Generate random values
Supplier<Double> rand = () -> Math.random();
System.out.println(rand.get());                          // e.g. 0.742...
▶ OUTPUT
Hello, World! 0.742... (varies each run)
⚠️
Consumer Has No compose()

Consumer only has andThen(), not compose(). This is intentional — since Consumer's return type is void, there's nothing to feed into a reverse chain. This is different from Function which has both.

07Bi-Variants & Unary/Binary Operators

All the core functional interfaces have "Bi" variants that accept two arguments. Two specialised interfaces, UnaryOperator and BinaryOperator, handle same-type I/O without repetitive generics.

BiVariants.java
import java.util.function.*;

// BiPredicate: two args, returns boolean
BiPredicate<Integer, Integer> sumTen = (a, b) -> (a + b) == 10;
System.out.println(sumTen.test(5, 5));           // true

// BiFunction: two args of possibly different types, one result
BiFunction<String, String, Integer> totalLen = (x, y) -> (x + y).length();
System.out.println(totalLen.apply("abc", ""));   // 3

// BiConsumer: two args, void return
BiConsumer<String, Integer> pair = (name, age) ->
    System.out.println(name + " is " + age);
pair.accept("Alice", 30);                          // Alice is 30

// UnaryOperator<T>: input and output are SAME type (extends Function<T,T>)
UnaryOperator<Integer> addOne = x -> x + 1;
System.out.println(addOne.apply(5));              // 6

// BinaryOperator<T>: two same-type inputs, same-type output
BinaryOperator<Integer> sum = (x, y) -> x + y;
System.out.println(sum.apply(4, 6));               // 10
▶ OUTPUT
true 3 Alice is 30 6 10
✅
When to Use Each

Use UnaryOperator instead of Function<T,T> when input and output types are the same — less boilerplate, more expressive. Use BinaryOperator instead of BiFunction<T,T,T>. Both are used heavily inside the Streams API: Stream.iterate uses UnaryOperator; reduce() uses BinaryOperator.

08Method References & Constructor References

A method reference is a shorthand for a lambda that only calls an existing method. The double-colon :: operator passes a reference to a method without invoking it — just as you pass a variable reference.

TypeSyntaxEquivalent LambdaExample
Static methodClass::staticMethodx -> Class.staticMethod(x)Integer::parseInt
Instance method (specific obj)obj::instanceMethodx -> obj.instanceMethod(x)System.out::println
Instance method (arbitrary obj)Class::instanceMethodx -> x.instanceMethod()String::toUpperCase
ConstructorClass::newx -> new Class(x)MobilePhone::new
MethodRefDemo.java
List<String> students = Arrays.asList("Ram", "Shyam", "Ghanshyam");

// Lambda form (verbose)
students.forEach(s -> System.out.println(s));

// Method reference form (clean) — passes println as reference
students.forEach(System.out::println);        // same output, less boilerplate

// Arbitrary instance method reference
List<String> upper = students.stream()
    .map(String::toUpperCase)              // calls toUpperCase() on each String
    .collect(Collectors.toList());
System.out.println(upper);                  // [RAM, SHYAM, GHANSHYAM]

// Constructor Reference
class Phone {
    String name;
    Phone(String name) { this.name = name; }  // constructor matching String -> Phone
}
List<Phone> phones = Arrays.asList("iPhone","Samsung").stream()
    .map(Phone::new)                         // new Phone(name) for each string
    .collect(Collectors.toList());
▶ OUTPUT
Ram Shyam Ghanshyam [RAM, SHYAM, GHANSHYAM]
⚠️
When NOT to Use Method References

Only use a method reference when the lambda does nothing but call a single method. If you need conditional logic, multiple statements, or chained calls (e.g. x -> x.toUpperCase().trim()), use a full lambda — a method reference won't work there.

09Introduction to Streams
What is a Stream?

A Stream is a sequence of elements supporting a functional processing pipeline. Streams do not store data — they are wrappers around a data source (like a List) that allow you to express complex data-processing logic concisely. They support lazy evaluation, method references, and all the functional interfaces we studied.

StageWhat It DoesExamplesReturns
SourceCreates the streamlist.stream(), Arrays.stream(arr)Stream<T>
IntermediateLazy transform; returns another streamfilter, map, sorted, flatMap, distinct, limit, skip, peekStream<T>
TerminalEager; triggers pipeline; closes streamcollect, forEach, count, reduce, findFirst, anyMatchResult or void
TraditionalVsStream.java
List<Integer> numbers = Arrays.asList(1,2,3,4,5);

// Traditional: 5 lines
int count = 0;
for (int n : numbers) { if (n % 2 == 0) count++; }
System.out.println(count);                     // 2

// Stream: one line
long c2 = numbers.stream()                       // source
    .filter(n -> n % 2 == 0)                     // intermediate: keep evens
    .count();                                     // terminal: count them
System.out.println(c2);                          // 2
▶ OUTPUT
2 2
💡
Stream is an Interface

Stream<T> is in java.util.stream. A stream can only be consumed once — after a terminal operation it is closed. Reusing it throws IllegalStateException.

10Creating Streams (7 Ways)
CreatingStreams.java
// 1. From Collection (most common)
List<String> list = Arrays.asList("a","b","c");
Stream<String> s1 = list.stream();

// 2. From Array
Stream<String> s2 = Arrays.stream(new String[]{"x","y"});

// 3. Stream.of() — from values directly
Stream<Integer> s3 = Stream.of(1,2,3,4,5);

// 4. Stream.generate(Supplier) — INFINITE; each element independently generated
Stream.generate(Math::random).limit(3).forEach(System.out::println);

// 5. Stream.iterate(seed, UnaryOperator) — each element derived from previous
Stream.iterate(1, x -> x + 1).limit(5).forEach(System.out::println); // 1 2 3 4 5

// 6. String.chars() — returns IntStream of char ASCII values
"hello".chars().forEach(System.out::println);  // 104 101 108 108 111

// 7. IntStream.range / rangeClosed
IntStream.range(1, 5).forEach(System.out::println);       // 1 2 3 4 (end exclusive)
IntStream.rangeClosed(1, 5).forEach(System.out::println); // 1 2 3 4 5 (end inclusive)
⭐
generate() vs iterate()

generate(Supplier): each element is independently produced; no relationship between elements. Good for constants or random values. iterate(seed, UnaryOperator): each element is computed from the previous; creates a mathematical sequence. Both are infinite — always add .limit(n).

11Intermediate Operations

Intermediate operations transform a Stream into another Stream. They are lazy — they don't execute until a terminal operation is called. You can chain as many as you like.

filter — Keep Matching Elements
FilterDemo.java
List<String> names = Arrays.asList("Akshit","Ram","Shyam","Alice","Bob");

// filter + collect: names starting with A, capped at 10
List<String> result = names.stream()
    .filter(n -> n.startsWith("A"))    // keep only names starting with A
    .limit(10)                           // cap at 10 (short-circuit on large input)
    .collect(Collectors.toList());
System.out.println(result);             // [Akshit, Alice]
▶ OUTPUT
[Akshit, Alice]
map — Transform Each Element
MapDemo.java
List<Integer> nums = Arrays.asList(2,3,5,9);

// Square and sort
List<Integer> squares = nums.stream()
    .map(n -> n * n)                     // 4, 9, 25, 81
    .sorted()                             // sort in natural order
    .collect(Collectors.toList());
System.out.println(squares);             // [4, 9, 25, 81]
▶ OUTPUT
[4, 9, 25, 81]
flatMap — Flatten Nested Collections

map() is one-to-one. flatMap() handles one-to-many — when each element is itself a collection. It maps each element to a Stream, then flattens all streams into one.

FlatMapDemo.java
// Flatten list of lists
List<List<String>> nested = Arrays.asList(
    Arrays.asList("Apple","Banana"),
    Arrays.asList("Cherry","Date")
);
List<String> flat = nested.stream()
    .flatMap(inner -> inner.stream())    // each inner list becomes its own stream
    .map(String::toUpperCase)
    .collect(Collectors.toList());
System.out.println(flat);                 // [APPLE, BANANA, CHERRY, DATE]

// Split sentences into individual words
List<String> sentences = Arrays.asList("Hello World", "Java Streams");
List<String> words = sentences.stream()
    .flatMap(s -> Arrays.stream(s.split(" ")))  // split each sentence on space
    .collect(Collectors.toList());
System.out.println(words);                 // [Hello, World, Java, Streams]
▶ OUTPUT
[APPLE, BANANA, CHERRY, DATE] [Hello, World, Java, Streams]
sorted, distinct, limit, skip, peek
OtherIntermediates.java
List<String> names = Arrays.asList("Charlie","Akshit","Alice","Bob","Akshit");

// sorted(): natural ordering
names.stream().sorted().forEach(System.out::println);

// sorted(Comparator): custom by string length
names.stream().sorted((a,b) -> a.length() - b.length())
    .forEach(System.out::println);

// distinct(): remove duplicates (uses .equals())
names.stream().distinct().forEach(System.out::println); // no duplicate Akshit

// skip(n): discard first n elements
Stream.iterate(1, x -> x+1).skip(10).limit(5)
    .forEach(System.out::println);  // 11 12 13 14 15

// peek(): intermediate side-effect; great for debugging
names.stream()
    .filter(n -> n.length() > 4)
    .peek(n -> System.out.println("Passed filter: " + n))
    .map(String::toUpperCase)
    .collect(Collectors.toList());
⭐
Stateful vs Stateless Operations

Stateless (filter, map, peek): process each element independently; optimal for parallel streams. Stateful (sorted, distinct, limit): must see multiple elements before deciding — sorted() must see ALL elements. Stateful operations are less efficient in parallel mode.

12Terminal Operations

Terminal operations consume the stream and produce a result. After a terminal operation, the stream is closed — reusing it throws IllegalStateException.

TerminalOps.java
List<Integer> nums = Arrays.asList(1,2,3,4,5);

// collect() — materialise into a collection
List<Integer> list = nums.stream().collect(Collectors.toList());

// forEach() — side effect per element (terminal)
nums.stream().forEach(System.out::println);

// count() — number of elements
long c = nums.stream().filter(n -> n % 2 == 0).count();  // 2

// reduce() — fold elements to a single value
Optional<Integer> sum = nums.stream().reduce((a,b) -> a + b);
System.out.println(sum.get());                           // 15

// reduce with identity (no Optional needed)
int sum2 = nums.stream().reduce(0, Integer::sum);        // 15

// anyMatch / allMatch / noneMatch — short-circuit boolean checks
boolean hasEven = nums.stream().anyMatch(n -> n%2==0);   // true
boolean allPos  = nums.stream().allMatch(n -> n>0);     // true
boolean noneNeg = nums.stream().noneMatch(n -> n<0);   // true

// findFirst() — first element in encounter order (returns Optional)
Optional<Integer> first = nums.stream().filter(n -> n>3).findFirst();
System.out.println(first.get());                          // 4

// min() / max() — uses Comparator
Optional<Integer> max = nums.stream().max(Comparator.naturalOrder());
System.out.println(max.get());                            // 5
▶ OUTPUT
15 15 true true true 4 5
forEachOrdered — Preserve Order in Parallel Streams
ForEachOrdered.java
List<Integer> list = Arrays.asList(1,2,3,4,5);

// forEach with parallel: arbitrary order (threads race)
list.parallelStream().forEach(System.out::println);      // 3 1 5 2 4 (varies)

// forEachOrdered: encounter order preserved even in parallel
list.parallelStream().forEachOrdered(System.out::println); // 1 2 3 4 5 always
⚠️
Stream Cannot Be Reused!

After any terminal operation the stream is closed. Storing a stream in a variable and reusing it throws IllegalStateException: stream has already been operated upon or closed. Always call list.stream() fresh each time.

13Lazy Evaluation & Stateful vs Stateless

Intermediate operations build a recipe; they execute nothing. Only when a terminal operation is called does the pipeline run — and only as far as needed (short-circuit operations stop early).

LazyEval.java
List<String> names = Arrays.asList("Anna","Bob","Charlie","David");

// Build pipeline — NOTHING executes yet
Stream<String> pipe = names.stream()
    .filter(n -> {
        System.out.println("Filtering: " + n);  // lazy: not called yet
        return n.length() > 3;
    });

System.out.println("-- Before terminal --");  // prints FIRST

// Terminal operation: NOW the filter lambda executes
long count = pipe.count();
System.out.println("count = " + count);
▶ OUTPUT
-- Before terminal -- Filtering: Anna Filtering: Bob Filtering: Charlie Filtering: David count = 3
💡
Why Laziness Matters in Practice

For findFirst() or limit(5), the pipeline stops as soon as the result is determined. Finding the first even number in a million-element stream stops at the very first even number found — not a million iterations. This makes stream pipelines extremely efficient.

14Parallel Streams

A parallel stream splits the workload across multiple threads using the Fork/Join framework. Use it for large, CPU-intensive, independent operations. Call list.parallelStream() or add .parallel() to any stream.

ParallelStream.java
// Sequential — one thread
long t1 = System.currentTimeMillis();
numbers.stream().map(Demo::factorial).collect(Collectors.toList());
System.out.println("Sequential: " + (System.currentTimeMillis()-t1) + "ms"); // ~150ms

// Parallel — multiple threads work simultaneously
long t2 = System.currentTimeMillis();
numbers.parallelStream().map(Demo::factorial).collect(Collectors.toList());
System.out.println("Parallel:   " + (System.currentTimeMillis()-t2) + "ms"); // ~17ms

// Convert parallel back to sequential when needed
numbers.parallelStream().sequential().map(Demo::factorial);
▶ OUTPUT
Sequential: ~150ms Parallel: ~17ms (hardware dependent)
Thread Safety: Shared Mutable State is Dangerous
ParallelSafety.java
// WRONG: race condition — result is unpredictable
int[] sum = {0};
IntStream.rangeClosed(1,100).parallel()
    .forEach(n -> sum[0] += n);        // NOT thread-safe!

// CORRECT option 1: AtomicInteger
AtomicInteger safe = new AtomicInteger(0);
IntStream.rangeClosed(1,100).parallel()
    .forEach(safe::addAndGet);
System.out.println(safe.get());         // 5050

// BEST option: reduce (no shared state at all)
int result = IntStream.rangeClosed(1,100).parallel()
    .reduce(0, Integer::sum);
System.out.println(result);              // 5050 always
▶ OUTPUT
??? (wrong, varies) 5050 5050
✅
When to Use Parallel Streams

Use when: dataset is large (10,000+ elements), operation is CPU-intensive, and elements are independent (each element's result does not depend on any other). Avoid for: small datasets (overhead outweighs benefit), I/O-bound tasks, and order-dependent computations like cumulative sums.

15Collectors — Complete Guide

The Collectors utility class provides factory methods for the most common collection strategies. A Collector defines how to accumulate stream elements into a final result.

CollectorsGuide.java
List<String> names = Arrays.asList("Alice","Bob","Alice","Charlie");
List<Integer> nums  = Arrays.asList(1,2,3,4,5);

// toList / toSet / toCollection
List<String> asList   = names.stream().collect(Collectors.toList());
Set<String>  asSet    = names.stream().collect(Collectors.toSet());  // deduped
LinkedList<String> linked = names.stream()
    .collect(Collectors.toCollection(LinkedList::new));

// joining: concatenate with separator, prefix, suffix
String joined = names.stream().distinct()
    .collect(Collectors.joining(", ", "[", "]"));   // [Alice, Bob, Charlie]

// groupingBy (1 arg): group by classifier, values are lists
Map<Integer, List<String>> byLen = names.stream()
    .collect(Collectors.groupingBy(String::length)); // {3=[Bob], 5=[Alice, Alice], 7=[Charlie]}

// groupingBy (2 args): with downstream collector
Map<Integer, Long> countByLen = names.stream()
    .collect(Collectors.groupingBy(String::length, Collectors.counting()));

// groupingBy (3 args): with Map factory (e.g. TreeMap for sorted keys)
Map<Integer, List<String>> sortedMap = names.stream()
    .collect(Collectors.groupingBy(String::length, TreeMap::new, Collectors.toList()));

// partitioningBy: always two groups — true and false
Map<Boolean, List<String>> parts = names.stream()
    .collect(Collectors.partitioningBy(n -> n.length() > 3));

// summarizingInt: count, sum, min, max, average at once
IntSummaryStatistics stats = nums.stream()
    .collect(Collectors.summarizingInt(x -> x));
System.out.println(stats.getCount()+" sum="+stats.getSum()+" avg="+stats.getAverage());

// toMap with merge function (REQUIRED when duplicates possible)
List<String> words = Arrays.asList("apple","banana","apple");
Map<String,Long> freq = words.stream()
    .collect(Collectors.toMap(
        w -> w,             // key: the word
        w -> 1L,            // initial value
        (old, n) -> old+n  // merge: add when duplicate key
    ));
System.out.println(freq);     // {banana=1, apple=2}

// mapping: transform elements BEFORE collecting (mini map inside a groupingBy)
// Collectors.mapping(mapper, downstream)
Map<Integer, List<String>> upperByLen = names.stream()
    .collect(Collectors.groupingBy(String::length,
        Collectors.mapping(String::toUpperCase, Collectors.toList())));
▶ OUTPUT
5 sum=15 avg=3.0 {banana=1, apple=2}
⚠️
toMap() Duplicate Key Exception

Without a merge function, toMap() throws IllegalStateException: Duplicate key if two elements map to the same key. Always provide the third argument when duplicates are possible.

16Primitive Streams — IntStream, LongStream, DoubleStream

Using Stream<Integer> boxes every int into an Integer object — unnecessary heap allocation. Java provides specialised primitive streams that avoid boxing and offer numeric-specific operations directly: sum(), average(), min(), max(), summaryStatistics().

PrimitiveStreams.java
// range (end exclusive) and rangeClosed (end inclusive)
IntStream.range(1,5).forEach(System.out::println);         // 1 2 3 4
IntStream.rangeClosed(1,5).forEach(System.out::println);   // 1 2 3 4 5

// Numeric aggregate operations — no boxing
int    sum = IntStream.rangeClosed(1,10).sum();
double avg = IntStream.rangeClosed(1,10).average().getAsDouble();
System.out.println(sum + " / " + avg);                   // 55 / 5.5

// summaryStatistics: all stats at once
IntSummaryStatistics stats = IntStream.rangeClosed(1,10).summaryStatistics();
System.out.println(stats);                               // count=10, sum=55, min=1, average=5.5, max=10

// boxing: IntStream -> Stream<Integer>
Stream<Integer> boxed = IntStream.range(1,5).boxed();

// mapToInt: Stream<T> -> IntStream (most common conversion)
List<String> ws = Arrays.asList("Hi","Hello","Hey");
int totalLen = ws.stream().mapToInt(String::length).sum();  // 2+5+3 = 10

// Random streams — 5 random ints between 1 and 99
new Random().ints(5, 1, 100).forEach(System.out::println);
▶ OUTPUT
55 / 5.5 IntSummaryStatistics{count=10, sum=55, min=1, average=5.500000, max=10} 10
💡
mapToInt is Your Friend

Whenever you have Stream<SomeObject> and need to compute numeric aggregates (sum, average, etc.), use .mapToInt(obj -> obj.getScore()) to get an IntStream and then call .sum() or .summaryStatistics() directly. Much cleaner than boxing to Integer and using Collectors.summingInt.

Filter Strings by Length and Uppercase

Given ["apple","kiwi","banana","fig","mango","plum"], collect all fruits whose name length is greater than 4, converted to upper case.

List<String> fruits = Arrays.asList("apple","kiwi","banana","fig","mango","plum");

Exercise 6 — Intermediate
Find the Longest Word
Concepts: reduce, max with Comparator, Optional

From ["cat","elephant","dog","hippopotamus","ant"], find the longest word.

Ex6.java
List<String> words = Arrays.asList("cat","elephant","dog","hippopotamus","ant");

// Using max() with Comparator
words.stream()
    .max(Comparator.comparingInt(String::length))  // element with max length
    .ifPresent(System.out::println);               // hippopotamus

// Using reduce (keeps the longer of each pair)
words.stream()
    .reduce((a, b) -> a.length() >= b.length() ? a : b)
    .ifPresent(System.out::println);               // hippopotamus

hippopotamus hippopotamus
Why it works: Both return Optional to safely handle empty streams. max() is more readable; reduce() is more flexible for custom accumulation logic.
Exercise 7 — Advanced
Group Employees by Department, Collect Names Only
Concepts: groupingBy + Collectors.mapping downstream

Given employees with name, dept, and salary, group by department and collect just the names per department (not full Employee objects).

Ex7.java
record Employee(String name, String dept, double salary) {}

List<Employee> emps = List.of(
    new Employee("Alice","Eng",90000), new Employee("Bob","Mkt",70000),
    new Employee("Charlie","Eng",85000), new Employee("Diana","Mkt",75000)
);

Map<String,List<String>> byDept = emps.stream()
    .collect(Collectors.groupingBy(
        Employee::dept,                         // group by department
        Collectors.mapping(                      // transform inside the group
            Employee::name,                      // extract just the name
            Collectors.toList()                  // collect names into a list
        )
    ));
byDept.forEach((d,n) -> System.out.println(d+": "+n));
▶ OUTPUT
Eng: [Alice, Charlie] Mkt: [Bob, Diana]
Why it works: Collectors.mapping() applies a transformation inside a downstream collector — like a map() step nested inside groupingBy, avoiding a separate map() intermediate operation.
Exercise 8 — Advanced
Top 3 Highest-Paid Employees
Concepts: sorted (reversed Comparator), limit, map, collect

From the same employee list, extract the names of the 3 highest-paid employees in descending order.

Ex8.java
List<String> top3 = emps.stream()
    .sorted(Comparator.comparingDouble(Employee::salary).reversed()) // highest first
    .limit(3)                                  // take first 3
    .map(Employee::name)                       // extract name
    .collect(Collectors.toList());
System.out.println(top3);
▶ OUTPUT
[Alice, Diana, Charlie]
Why it works: reversed() flips the Comparator so highest salary comes first. limit(3) short-circuits the pipeline — efficient even on large lists. map() extracts names after sorting.
18Interview Questions (10 Q&A)
Q1. What is a Stream in Java 8? How is it different from a Collection?

A Stream is a sequence of elements supporting aggregate operations in a declarative, functional style. A Collection stores data; a Stream processes data. Streams are lazy (nothing runs until a terminal op), can be infinite, and can only be consumed once. Collections can be iterated multiple times and hold the data persistently in memory.

⭐
Asked Very Frequently

This is asked in virtually every Java backend interview. Know this distinction cold: Collection = storage, Stream = computation pipeline.

Q2. What is the difference between intermediate and terminal operations?

Intermediate operations (filter, map, sorted) are lazy — they return a new Stream and execute nothing until a terminal operation is called. Terminal operations (collect, count, forEach) are eager — they trigger the entire pipeline and produce a final result or side effect. After a terminal operation, the stream is closed and cannot be reused.

⭐
Asked Very Frequently

Follow-up: "Name 5 intermediate and 5 terminal operations." Common answer: Intermediate — filter, map, flatMap, sorted, distinct. Terminal — collect, forEach, count, reduce, findFirst.

Q3. What is a Functional Interface? Give built-in examples.

A functional interface has exactly one abstract method. The @FunctionalInterface annotation is optional but enforces this at compile time. Built-in examples in java.util.function: Predicate<T> (test), Function<T,R> (apply), Consumer<T> (accept), Supplier<T> (get). From JDK: Runnable, Callable, Comparator.

Q4. What is the difference between map() and flatMap()?

map() is one-to-one — each element produces exactly one result. flatMap() is one-to-many — each element produces a Stream, and all those streams are flattened into one. Use flatMap when the mapping function returns a Collection or Stream per element (e.g., splitting sentences into words, or getting orders from a list of customers).

MapVsFlatMap.java
// map: Stream<String> -> Stream<Integer> (1-to-1)
names.stream().map(String::length);

// flatMap: Stream<List<String>> -> Stream<String> (1-to-many, then flattened)
nested.stream().flatMap(Collection::stream);
⭐
Tricky Follow-up

"What happens if the mapper in flatMap returns null?" — It throws NullPointerException. Always return Stream.empty() instead of null: x -> x == null ? Stream.empty() : x.stream().

Q5. What is lazy evaluation in streams and why does it matter?

Intermediate operations are not executed until a terminal operation is invoked. This allows the JVM to optimise the pipeline — short-circuit operations (findFirst, limit, anyMatch) can terminate early without processing all elements. For infinite streams, laziness is what prevents infinite loops.

Q6. What is the difference between findFirst() and findAny()?

findFirst() returns the first element in encounter order — deterministic for both sequential and parallel streams. findAny() may return any element and is non-deterministic in parallel streams (whichever thread finishes first), but it is faster because threads don't need to coordinate on ordering.

Q7. Why can't you reuse a stream after a terminal operation?

A Stream is a one-pass abstraction — it does not store data. After a terminal operation the stream is closed (all internal resources released). Reusing it throws IllegalStateException: stream has already been operated upon or closed. The fix is simple: call list.stream() fresh each time you need a new stream from the same source.

Q8. What are the risks of using parallel streams with shared mutable state?

Multiple threads updating the same variable simultaneously create race conditions — the result is wrong and non-deterministic. Lambda variables must be effectively final (compiler enforcement). Use thread-safe alternatives like AtomicInteger, or better: avoid shared state entirely by using reduce() or collect() which accumulate results without shared mutation.

SafeParallel.java
// WRONG — race condition, wrong answer
int[] sum = {0};
stream.parallel().forEach(n -> sum[0] += n);

// BEST — no shared state, always correct
int result = IntStream.rangeClosed(1,100).parallel().reduce(0, Integer::sum);
Q9. What is the difference between Collectors.groupingBy() and partitioningBy()?

groupingBy() groups elements into a Map<K, List<T>> — the key can be any type and the number of groups is variable. partitioningBy() always produces exactly two groups (Map<Boolean, List<T>>) based on a Predicate — a true group and a false group. Use partitioningBy when the criterion is strictly binary.

⭐
Asked Very Frequently

Key insight: partitioningBy always creates both groups (even if empty). groupingBy only creates groups that have at least one member.

Q10. Output-based: What does this code print? (Trace through carefully)
OutputQ.java
List<String> names = Arrays.asList("Bob","Alice","Charlie","Dan");
String r = names.stream()
    .filter(n -> n.length() > 4)    // step 1: keep >4 chars
    .map(String::toUpperCase)        // step 2: upper case
    .skip(1)                          // step 3: skip first result
    .findFirst()                       // step 4: return first remaining
    .orElse("none");
System.out.println(r);
▶ OUTPUT
CHARLIE

Trace: filter keeps [Alice(5), Charlie(7)]; map produces [ALICE, CHARLIE]; skip(1) drops ALICE; findFirst() returns CHARLIE. Bob(3) and Dan(3) are filtered out immediately.

⭐
Asked Very Frequently

Important insight: the pipeline is NOT stage-by-stage (first all filter, then all map). It is element-by-element. Each element flows through all stages before the next element is processed. Bob tries filter → fails → done. Alice tries filter → passes → uppercased ALICE → skip drops it. Charlie tries filter → passes → CHARLIE → returned by findFirst.

Common Pitfalls
⚠️
Pitfall 1: Stream Reuse

After any terminal operation, the stream is closed. Storing a stream in a variable and using it twice throws IllegalStateException. Fix: always call list.stream() fresh each time.

⚠️
Pitfall 2: Infinite Stream Without limit()

Stream.generate() and Stream.iterate() produce infinite streams. Calling a non-short-circuit terminal op like count() hangs forever (OOM). Always pair with .limit(n).

⚠️
Pitfall 3: toMap() Duplicate Keys

Without a merge function, toMap() throws IllegalStateException: Duplicate key. Always provide the third argument when duplicates are possible: Collectors.toMap(k, v, (old, n) -> old).

⚠️
Pitfall 4: Parallel Stream with Shared Mutable State

Updating a shared variable inside a parallel stream causes race conditions. Use reduce(), collect(), or AtomicInteger. Lambda captured variables must be final or effectively final.

⚠️
Pitfall 5: Parallel for Small Data

Thread management overhead makes parallelStream() slower than sequential for small datasets or trivial operations. Only use parallel when dataset is large AND per-element work is CPU-intensive AND elements are independent.

5 Golden Tips
✅
Remember These Always

1. Think in 3 stages: SOURCE → INTERMEDIATE OPS → TERMINAL OP.
2. Intermediate ops are LAZY — nothing runs until a terminal op is called.
3. A stream is ONE-TIME-USE — recreate it from the source when needed again.
4. Prefer primitive streams (IntStream) for numeric work to avoid boxing overhead.
5. Use parallelStream() only for large, CPU-intensive, independent operations.
