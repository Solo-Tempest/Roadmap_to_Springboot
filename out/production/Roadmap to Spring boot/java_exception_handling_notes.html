Section 01

Types of Errors in Java

The Three Types of Errors

Before understanding exceptions, you must know that Java programs can fail in three fundamentally different ways. Each type has a different cause, different detection time, and a different fix strategy.

Error TypeWhen DetectedWhose FaultDifficulty to Fix
Syntax ErrorCompile timeProgrammerEasy — compiler points to it
Logical ErrorRun time (wrong output)ProgrammerHard — requires testing
Runtime ErrorRun time (program crashes)Environment / InputMedium — must be handled

1. Syntax Errors (Compile-time Errors)

A syntax error means the code is grammatically wrong — Java cannot even understand it. Your IDE underlines it in red. The compiler will refuse to compile until it is fixed.

SyntaxDemo.java
public class SyntaxDemo {
    public static void main(String[] args) {
        // Missing semicolon — SYNTAX ERROR
        System.out.println("Hello")   // ❌ will NOT compile

        // Capital P — SYNTAX ERROR (wrong method name)
        System.out.Println("Hello");  // ❌ cannot find symbol
    }
}
Compiler Output
error: ';' expected error: cannot find symbol — Println
✅
Why these are easy to fix

The compiler tells you exactly which line is wrong. Your IDE highlights it in real time even before you compile. Just read the error message and fix the syntax.

2. Logical Errors (Bugs)

A logical error means the code compiles and runs perfectly — but produces the wrong answer. You were asked to multiply, but you divided. You were asked to add, but you subtracted. The computer did exactly what you told it — the problem is what you told it.

LogicalDemo.java
public class LogicalDemo {
    static int divide(int a, int b) {
        return a * b;  // ❌ should be a / b — LOGICAL ERROR
    }

    public static void main(String[] args) {
        System.out.println(divide(10, 2));  // Prints 20, expected 5
    }
}
Output (wrong!)
20 ← expected 5, no error shown
⚠️
Why these are hardest to fix

The program doesn't crash — it gives you a wrong answer silently. Finding logical errors requires thorough testing with many different inputs and careful comparison of expected vs actual output. These are called bugs.

3. Runtime Errors — Why They Matter Most

A runtime error is the most dangerous kind. The code compiles fine, the logic is correct — but at some point during execution, something unexpected happens and the program crashes mid-run.

Real-world analogy from the instructor: You built an application that opens a file from your C drive. It worked perfectly for months. One night you delete that file. Next morning the application crashes the moment it tries to open it — a runtime error.

RuntimeDemo.java
public class RuntimeDemo {
    public static void main(String[] args) {
        int[] numerators   = {10, 200, 30, 40};
        int[] denominators = {1,   2,   0,  4};  // 0 causes crash!

        for (int i = 0; i < numerators.length; i++) {
            System.out.println(numerators[i] / denominators[i]);
        }
        System.out.println("Good job! ✅");  // Never reached!
    }
}
Output
10 100 Exception in thread "main" java.lang.ArithmeticException: / by zero at RuntimeDemo.main(RuntimeDemo.java:6) ← "Good job!" was never printed — program stopped!
💡
Why runtime errors are dangerous

The program stopped at 30 / 0 — the perfectly valid 40 / 4 never ran. In critical systems this is catastrophic: medical scanning machines, banking software, flight control systems. You cannot afford a mid-execution crash. This is why exception handling exists — not to prevent errors, but to keep your program running gracefully when they occur.

✅ Do
  • Test your code with extreme inputs (zero, null, empty)
  • Handle runtime errors so the program can continue
  • Use an IDE — it catches syntax errors in real time
  • Write unit tests to catch logical errors early
❌ Don't
  • Assume the environment will always be perfect
  • Ship code without testing edge cases
  • Ignore compiler warnings
  • Let a runtime error crash the entire application
Section 02

What is an Exception?

Formal Definition

An exception is an event that disrupts the normal flow of the program. It is an object that is created and thrown by the JVM at runtime when something goes wrong.

📌
Key Points to Remember

• Exceptions are runtime errors — not compile-time.
• An exception is an object (instance of a class like ArithmeticException).
• When an exception occurs, JVM creates that object and throws it.
• If nobody catches it, the program terminates abnormally.

How the JVM Creates an Exception Object

When 30 / 0 happens, the JVM doesn't just display an error message — it actually creates an object of class ArithmeticException, fills it with details (message, stack trace), and throws it. This is why you can catch it with a variable like e — you are receiving that object.

ExceptionObject.java
// When JVM encounters divide-by-zero, it internally does something like:
throw new ArithmeticException("/ by zero");
// This object is what you receive in your catch block as 'e'

// The object has these useful methods:
e.getMessage()      // → "/ by zero"
e.toString()        // → "java.lang.ArithmeticException: / by zero"
e.printStackTrace() // → full stack trace to console
e.getStackTrace()   // → array of StackTraceElement objects

Exceptions vs. Errors — Quick Distinction

Not everything that goes wrong is an exception in the Java sense. Java draws a clear line:

CategoryClassCan You Handle It?Example
ExceptionException✅ Yes — use try-catchArithmeticException, IOException
ErrorError❌ No — JVM handles itOutOfMemoryError, StackOverflowError
⚠️
Exceptions you handle. Errors you can't.

If the JVM runs out of memory, there is nothing your code can do. These Error subclasses are catastrophic JVM-level failures. Focus your exception handling on Exception and its subclasses.

Why Exceptions Must Be Handled

The instructor's analogy: Imagine you're in a game battle where winning earns you $1,000,000 — and suddenly the game crashes. Or imagine a patient inside a medical scanner when the software stops. These scenarios make it crystal clear: letting a runtime error terminate your application is unacceptable in any serious system.

Exception handling doesn't eliminate exceptions — it ensures the program keeps running gracefully even when they occur. The code that should have run after the exception still runs.

⭐
Interview Tip

"Exception handling is a mechanism to handle runtime errors so that the normal flow of the application can be maintained." — This is the standard definition. Know it word for word.

Section 03

try-catch-finally

The Core Idea — Normal vs. Critical Statements

The instructor's analogy: "Imagine you're walking with a child in a mall. If the child runs away, you calmly walk after them — no panic. But if the same thing happens on a busy highway, you leap to grab the child immediately. The situation is critical."

In the same way, Java statements are either normal (nothing can really go wrong) or critical (an exception might occur). Critical statements go inside a try block.

The try Block

You wrap risky/critical code inside a try block. You're telling Java: "Try to execute this. If it works, great. If not, don't stop the program — go to the catch block."

TryDemo.java
int i = 0;   // normal statement — nothing can go wrong
int j = 0;   // normal

try {
    j = 18 / i;   // CRITICAL — i could be 0!
    System.out.println("Result: " + j);
} catch (Exception e) {
    System.out.println("Something went wrong: " + e);
}
System.out.println("Bye!");  // This ALWAYS runs now
Output (when i = 0)
Something went wrong: java.lang.ArithmeticException: / by zero Bye!
Output (when i = 4)
Result: 4 Bye!

The catch Block

When an exception is thrown inside try, execution immediately jumps to catch. The exception object is passed to the catch parameter (e by convention). The catch block only executes if there is an exception — otherwise it is completely skipped.

💡
The catch parameter is just a variable

You can name it anything: e, ex, obj, exception. By convention, e is used. It holds a reference to the exception object that was thrown.

The finally Block

The finally block ALWAYS runs — whether an exception occurred or not. It is primarily used for resource cleanup: closing files, database connections, network streams.

FinallyDemo.java
try {
    int result = 10 / 0;   // throws ArithmeticException
    System.out.println("Result: " + result);
} catch (Exception e) {
    System.out.println("Exception caught: " + e.getMessage());
} finally {
    System.out.println("Finally block — ALWAYS runs!");
    // Close files, DB connections, streams here
}
Output
Exception caught: / by zero Finally block — ALWAYS runs!
FinallyNoException.java
try {
    int result = 10 / 2;   // no exception
    System.out.println("Result: " + result);
} catch (Exception e) {
    System.out.println("This won't run");
} finally {
    System.out.println("Finally block — ALWAYS runs!");
}
Output
Result: 5 Finally block — ALWAYS runs!

try-finally Without catch (Valid Syntax)

You do not always need a catch block. If you only want to guarantee cleanup without handling the exception, try-finally alone is valid. The exception will still propagate upward — you just ensure the cleanup runs first.

TryFinallyOnly.java
try {
    // risky code
    doSomethingRisky();
} finally {
    // always closes, even if exception propagates up
    connection.close();
}
// Note: no catch block — exception still propagates to caller
⭐
Interview Tip — finally execution

finally runs even if there is a return statement inside try or catch. The only case where finally doesn't run is System.exit() being called or a JVM crash.

✅ Do
  • Put only critical/risky code inside try
  • Use finally to close resources
  • Always handle or propagate exceptions
  • Print meaningful messages in catch
❌ Don't
  • Put all code inside try unnecessarily
  • Leave the catch block completely empty
  • Rely on try-catch to avoid fixing real bugs
  • Forget to close resources in finally
Section 04

Stack Trace & Exception Methods

What is a Stack Trace?

A stack trace is a snapshot of the call stack at the moment an exception occurred. It tells you exactly which method called which method, and on which line the exception happened. It reads from bottom (oldest call) to top (where the exception occurred).

Stack Trace Reading Guide
Exception in thread "main" java.lang.ArithmeticException: / by zero
    at LevelThree.methodC(LevelThree.java:5)    ← exception happened HERE
    at LevelTwo.methodB(LevelTwo.java:3)      ← called by this
    at LevelOne.methodA(LevelOne.java:3)      ← called by this
    at Main.main(Main.java:5)               ← execution started here

This trace shows: main called methodA → methodA called methodB → methodB called methodC → exception thrown in methodC on line 5.

Key Exception Methods

ExceptionMethods.java
try {
    int result = 10 / 0;
} catch (ArithmeticException e) {

    // 1. getMessage() — returns just the message string
    System.out.println(e.getMessage());
    // Output: / by zero

    // 2. toString() — class name + message (called by println)
    System.out.println(e.toString());
    // Output: java.lang.ArithmeticException: / by zero

    // 3. printStackTrace() — full trace to stderr
    e.printStackTrace();

    // 4. getStackTrace() — returns array you can loop over
    StackTraceElement[] st = e.getStackTrace();
    for (StackTraceElement frame : st) {
        System.out.println(frame);
    }
}
Output
/ by zero java.lang.ArithmeticException: / by zero java.lang.ArithmeticException: / by zero at ExceptionMethods.main(ExceptionMethods.java:3) ExceptionMethods.main(ExceptionMethods.java:3)
💡
Why println(e) shows the full class name

When you pass an object to System.out.println(), Java internally calls its toString() method. The Throwable class defines toString() to return ClassName: message. That's why printing e shows something like java.lang.ArithmeticException: / by zero.

Where the Message Comes From

When divide-by-zero happens, the JVM creates an ArithmeticException object and passes "/ by zero" as the message to its constructor. This message travels up through RuntimeException → Exception → Throwable, where it is stored in a field called detailMessage. The toString() method in Throwable then formats it as ClassName: detailMessage.

MethodReturnsUse Case
getMessage()Just the message stringDisplay user-friendly error
toString()ClassName: messageQuick debugging
printStackTrace()Full trace to stderrDebugging — finding root cause
getStackTrace()Array of StackTraceElementProgrammatic trace processing
getCause()The original exceptionException chaining
⭐
Interview Tip

printStackTrace() is great for debugging but should NOT be used in production code. Use a proper logging framework (Log4j, SLF4J) instead. Interviewers look for this awareness.

Section 05

Multiple catch Blocks

Why One catch Is Not Enough

A single try block can contain many statements. Different statements can throw different types of exceptions. If you use a single generic catch(Exception e), you cannot provide specific messages or handling for each case.

MultiCatchDemo.java
int i = 0;
int[] nums = new int[5];
String str = null;

try {
    int j = 18 / i;           // can throw ArithmeticException
    System.out.println(nums[5]);  // can throw ArrayIndexOutOfBoundsException
    System.out.println(str.length()); // can throw NullPointerException
}
catch (ArithmeticException e) {
    System.out.println("Cannot divide by zero!");
}
catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("Stay within array limits!");
}
catch (NullPointerException e) {
    System.out.println("Object is null!");
}
catch (Exception e) {
    // Safety net — catches anything else not handled above
    System.out.println("Something went wrong: " + e);
}
System.out.println("Bye!");
Output (when i = 0)
Cannot divide by zero! Bye!
Output (when i = 2)
Stay within array limits! Bye!
💡
Execution jumps immediately

The moment an exception occurs inside try, execution jumps to the matching catch block. The remaining statements inside try are skipped entirely. Only one catch block executes per exception.

The Golden Rule: Parent Class Must Come Last

Since Exception is the parent of all exceptions, it can catch everything. If you place it first, Java will use it for every exception and the specific catch blocks below it become unreachable — a compile error.

CatchOrder.java
// ❌ WRONG — Exception is parent, placed first
catch (Exception e) { ... }              // catches everything
catch (ArithmeticException e) { ... }   // COMPILE ERROR: unreachable!

// ✅ CORRECT — specific first, parent last
catch (ArithmeticException e) { ... }   // specific
catch (NullPointerException e) { ... }  // specific
catch (Exception e) { ... }              // safety net — always last
⭐
Interview Tip

"In multiple catch blocks, the most specific exception class must come first and the most general (parent) class must come last. Placing a parent class before a child class causes a compile-time error: unreachable catch block."

✅ Do
  • Write specific catch blocks for each expected exception
  • Always put Exception as a final safety net
  • Give meaningful messages per exception type
❌ Don't
  • Put parent class catch block before child class
  • Use a single generic catch when you can be specific
  • Print the same message for every exception type
Section 06

Multi-catch Block (Java 7+)

Catch Multiple Types in One Block

Java 7 introduced the ability to catch multiple exception types in a single catch block using the pipe | operator. This reduces code duplication when you want to handle different exceptions the same way.

MultiCatchSyntax.java
try {
    // code that might throw multiple exception types
    riskyMethod();
}
// Before Java 7 — repetitive:
catch (ArithmeticException e)           { log(e); return -1; }
catch (NullPointerException e)           { log(e); return -1; }
catch (ArrayIndexOutOfBoundsException e) { log(e); return -1; }

// Java 7+ — clean multi-catch:
catch (ArithmeticException | NullPointerException | ArrayIndexOutOfBoundsException e) {
    log(e);
    return -1;
}

The Disjoint Rule

In a multi-catch block, the exception types must be disjoint — meaning no type can be a subclass of another type in the same multi-catch. If they overlap, the compiler will reject it because the more specific one would always be matched by the parent.

DisjointRule.java
// ❌ COMPILE ERROR — RuntimeException is parent of ArithmeticException
catch (RuntimeException | ArithmeticException e) { }
// Error: Alternatives in a multi-catch statement cannot be related

// ✅ CORRECT — all three are siblings (disjoint)
catch (ArithmeticException | NullPointerException | IOException e) { }
⭐
Interview Tip

In a multi-catch block, the variable e is implicitly final — you cannot reassign it inside the block. This is by design in Java 7+.

Section 07

Nested try-catch

try Inside try

Just like nested if statements or nested loops, you can place a try-catch block inside another try block. This is useful when different parts of your code need different levels of error handling.

NestedTryCatch.java
try {
    System.out.println("Outer try — start");

    try {
        int[] arr = new int[5];
        arr[10] = 99;  // ArrayIndexOutOfBoundsException
    } catch (ArrayIndexOutOfBoundsException e) {
        System.out.println("Inner catch: " + e.getMessage());
    }

    System.out.println("Outer try — continues after inner try");
    int result = 10 / 0;  // ArithmeticException

} catch (ArithmeticException e) {
    System.out.println("Outer catch: " + e.getMessage());
}
Output
Outer try — start Inner catch: Index 10 out of bounds for length 5 Outer try — continues after inner try Outer catch: / by zero
💡
Key behaviour

If the inner catch handles the exception, execution continues in the outer try block normally. If the inner try throws an exception that its inner catch cannot handle, it propagates to the outer catch.

Section 08

Exception Hierarchy

The Full Tree

Java's exception system is built on a class hierarchy rooted at Object. Understanding this tree is essential — it determines which exceptions are related, which catch blocks can handle which exceptions, and which ones you are forced to handle.

Java Exception Hierarchy Tree
Object
└── Throwable                     ← parent of everything throwable
    ├── Error                     ← JVM-level failures (do NOT handle)
    │   ├── OutOfMemoryError
    │   ├── StackOverflowError
    │   ├── ThreadDeath
    │   └── VirtualMachineError
    │
    └── Exception                 ← handle these in your code
        ├── RuntimeException      ← Unchecked — optional to handle
        │   ├── ArithmeticException
        │   ├── NullPointerException
        │   ├── ArrayIndexOutOfBoundsException
        │   ├── ClassCastException
        │   └── NumberFormatException
        │
        ├── IOException           ← Checked — MUST handle or declare
        │   └── FileNotFoundException
        ├── SQLException          ← Checked
        └── ClassNotFoundException ← Checked

Throwable — The Root

Throwable is the superclass of both Error and Exception. It provides the core infrastructure: the detailMessage field, the cause field, and the methods getMessage(), toString(), printStackTrace(). Everything you throw in Java must be a Throwable.

⚠️
Throwable trivia

Most Java class names ending in -able are interfaces (Runnable, Serializable, Cloneable). Throwable is a rare exception — it's a class, not an interface. Don't catch Throwable in production code; always catch Exception at the broadest level.

Error — Don't Touch It

Error represents catastrophic JVM-level failures. You cannot prevent or recover from them in code. If the JVM runs out of memory — what are you going to do inside your catch block? There's nothing left to run with.

Error ClassWhen It Happens
OutOfMemoryErrorJVM cannot allocate more memory
StackOverflowErrorInfinite recursion exhausts the call stack
ThreadDeathA thread is killed by the JVM
VirtualMachineErrorInternal JVM failure

Using Hierarchy in catch Blocks

Because of inheritance, a catch block for a parent class will catch all its child classes too. This is why catch(Exception e) catches everything — and why parent class must always be placed last in multiple catch blocks.

HierarchyCatch.java
try {
    int x = 10 / 0;
}
// All of these can catch ArithmeticException:
catch (ArithmeticException e)  { } // most specific ✅
catch (RuntimeException e)     { } // parent — also works
catch (Exception e)            { } // grandparent — also works
catch (Throwable e)            { } // root — avoid in practice
⭐
Interview Tip — Polymorphism in Exception Handling

The reason a parent catch block catches child exceptions is polymorphism. An ArithmeticException IS-A RuntimeException IS-A Exception IS-A Throwable. This is a classic interview question connecting OOP concepts to exception handling.

Section 09

Checked vs. Unchecked Exceptions

The Core Distinction

Java divides exceptions into two groups based on whether the compiler forces you to handle them or not.

FeatureCheckedUnchecked
Also calledCompile-time exceptionsRuntime exceptions
Parent classException (excluding RuntimeException)RuntimeException
Compiler enforces?✅ Yes — must handle or declare❌ No — optional
ExamplesIOException, SQLException, FileNotFoundException, ClassNotFoundExceptionNullPointerException, ArithmeticException, ArrayIndexOutOfBoundsException

Unchecked Exceptions — Optional to Handle

The compiler doesn't complain if you don't handle these. They are usually caused by programming mistakes (accessing null, dividing by zero, going out of array bounds). You should handle them, but Java doesn't force you.

Unchecked.java
// Compiler is FINE with this — no forced handling
public static void main(String[] args) {
    String s = null;
    System.out.println(s.length()); // NullPointerException at runtime
    // Compiles fine — crashes at runtime
}

Checked Exceptions — Must Handle or Declare

If a method call can throw a checked exception, the compiler will not let your code compile until you either handle it with try-catch or declare it with throws. This forces you to think about error handling for operations that commonly fail (file not found, network down, database unavailable).

CheckedDemo.java
// FileReader throws FileNotFoundException — a CHECKED exception
// ❌ This does NOT compile without handling:
FileReader fr = new FileReader("a.txt"); // Compile error!

// ✅ Option 1: Handle with try-catch
try {
    FileReader fr = new FileReader("a.txt");
} catch (FileNotFoundException e) {
    System.out.println("File not found: " + e.getMessage());
}

// ✅ Option 2: Declare with throws (pass to caller)
public void readFile() throws FileNotFoundException {
    FileReader fr = new FileReader("a.txt");
}
💡
Why checked exceptions exist

For operations where failure is foreseeable and common — reading a file that might not exist, connecting to a database that might be down — Java forces you to acknowledge the possibility of failure. This is a design decision to improve reliability.

⭐
Interview Tip

"Checked exceptions are checked at compile time. The compiler forces you to handle or declare them. Unchecked exceptions (RuntimeException subclasses) are not checked at compile time — they occur at runtime and are typically caused by programming errors." — This is a very common interview question.

Section 10

Exception Propagation

How Exceptions Travel Up the Call Stack

When an exception is thrown in a method and that method doesn't handle it, the exception propagates (travels up) to the method that called it. If that method doesn't handle it either, it keeps going up until it either finds a handler or reaches main() — at which point the JVM terminates the program.

Propagation.java
public class Propagation {

    static void levelThree() {
        int[] arr = new int[5];
        arr[10] = 99;  // Exception thrown HERE — not handled
    }

    static void levelTwo() {
        levelThree();   // exception propagates here
    }

    static void levelOne() {
        levelTwo();     // exception propagates here
    }

    public static void main(String[] args) {
        try {
            levelOne();   // exception propagates here — CAUGHT!
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Caught in main: " + e.getMessage());
        }
    }
}
Output
Caught in main: Index 10 out of bounds for length 5
Exception Propagation Flow
main()  ──calls──▶  levelOne()  ──calls──▶  levelTwo()  ──calls──▶  levelThree()
  ▲                    ▲                   ▲                   💥 EXCEPTION
  │                    │                   │
  catches it ◀────────── propagates ◀──────── propagates ◀───────────────

What Happens When Nobody Catches It

If the exception reaches main() and still isn't handled, the JVM's default handler takes over: it prints the full stack trace to the console and terminates the program. Any code after the point of the exception is never executed.

⚠️
Unchecked exceptions propagate silently

Unlike checked exceptions, unchecked exceptions (RuntimeException subclasses) don't need to be declared with throws as they propagate. They travel up the call stack invisibly until they either find a handler or crash the program.

Section 11

throws Keyword — Ducking Exceptions

What throws Does

The throws keyword in a method signature is a warning label. It tells the caller: "This method may throw this exception — you deal with it." The method itself doesn't handle the exception — it delegates (ducks) responsibility to whoever calls it.

The instructor's analogy: "Some employees never take responsibility. When something goes wrong, they say 'Talk to my manager.' The throws keyword is exactly that — 'I'm not handling this, talk to my caller.'"

ThrowsDemo.java
class A {
    public void show() throws ClassNotFoundException {
        // Delegates — doesn't handle it itself
        Class.forName("calc");  // might throw ClassNotFoundException
    }
}

public class Main {
    public static void main(String[] args) {
        A obj = new A();
        try {
            obj.show();  // caller handles it here
        } catch (ClassNotFoundException e) {
            System.out.println("Not able to find the class: " + e.getMessage());
        }
    }
}
Output
Not able to find the class: calc

Chaining throws Up the Call Stack

You can chain throws declarations across multiple methods. Each method passes responsibility up until someone actually handles it with try-catch.

ThrowsChain.java
static void methodOne() throws FileNotFoundException {
    new FileReader("a.txt");  // doesn't handle it
}

static void methodTwo() throws FileNotFoundException {
    methodOne();  // passes it further up
}

public static void main(String[] args) {
    try {
        methodTwo();  // finally handled here
    } catch (FileNotFoundException e) {
        System.out.println("File not found!");
    }
}

Never throws from main()

Technically you can write public static void main(String[] args) throws Exception. But this is a bad practice. If main() throws, the JVM receives the exception — and the JVM's only response is to print the stack trace and terminate the program. You have gained nothing from exception handling.

⚠️
Never throw from main to JVM

If you propagate an exception all the way to main() without catching it, JVM handles it — which means program crash + stack trace dump. Always handle exceptions somewhere in the chain using try-catch.

⭐
throws is only required for Checked Exceptions

For unchecked exceptions (RuntimeException subclasses), adding throws to the method signature is optional and generally not done. For checked exceptions, it is mandatory if you are not handling them locally.

Section 12

throw Keyword

Manually Throwing an Exception

The throw keyword lets you create and throw an exception object yourself. You are not waiting for the JVM to detect an error — you are proactively raising one based on your own business logic.

ThrowDemo.java
int i = 20;
int j = 0;

try {
    j = 18 / i;

    // Even though no JVM error, we don't want 0 as result
    if (j == 0) {
        // Manually throw an exception with a custom message
        throw new ArithmeticException("I don't want to print 0");
    }
    System.out.println("Result: " + j);
} catch (ArithmeticException e) {
    System.out.println("Exception: " + e.getMessage());
    System.out.println("Default result: 18");
}
Output
Exception: I don't want to print 0 Default result: 18

Real-World Use Cases for throw

The real power of throw is enforcing business rules. Consider a primary-backup database scenario: if connecting to the primary database fails, you want to automatically switch to the backup — not just show an error to the user.

BusinessRule.java
public static int divide(int a, int b) {
    if (b == 0) {
        throw new ArithmeticException("Denominator cannot be zero!");
    }
    return a / b;
}
💡
throw vs. throws — The Difference

throw (no s) is used inside a method body to actually throw an exception object. throws (with s) is used in a method signature to declare that the method might throw an exception. One executes — the other declares.

Section 13

try-with-resources (Java 7+)

The Problem: Forgetting to Close Resources

Whenever you open a file, database connection, or network stream, you must close it when done. Forgetting wastes system resources. Even with finally, it's verbose and error-prone — especially when the close itself might throw an exception.

OldWay.java
// Old way — manual close in finally (verbose)
BufferedReader br = null;
try {
    br = new BufferedReader(new InputStreamReader(System.in));
    String line = br.readLine();
    System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) {
        try { br.close(); }              // close itself can throw!
        catch (IOException e) { }
    }
}

The Solution: try-with-resources

Declare the resource inside the parentheses of try. Java will automatically close the resource when the try block completes — whether it completed normally or with an exception. No finally needed.

TryWithResources.java
// New way — auto-close, clean syntax
try (BufferedReader br = new BufferedReader(new InputStreamReader(System.in))) {
    String line = br.readLine();
    System.out.println("You typed: " + line);
} catch (IOException e) {
    e.printStackTrace();
}
// br is automatically closed here — no finally needed!

// Multiple resources — both auto-closed:
try (FileReader fr = new FileReader("in.txt");
     FileWriter fw = new FileWriter("out.txt")) {
    // use fr and fw
} catch (IOException e) { ... }
// both fr and fw closed automatically

The AutoCloseable Interface

This works because BufferedReader implements AutoCloseable. Any class that implements AutoCloseable (which provides a close() method) can be used in try-with-resources. Java calls close() automatically when the block exits.

AutoCloseable Inheritance Chain
AutoCloseable   ← interface with close() method
└── Closeable   ← extends AutoCloseable
    └── Reader
        └── BufferedReader  ← implements AutoCloseable ✅

Other AutoCloseable implementors:
InputStream, OutputStream, FileReader, FileWriter,
Connection (JDBC), PreparedStatement, ResultSet
⭐
Interview Tip

try-with-resources was introduced in Java 7. Resources are closed in reverse order of declaration (LIFO). The close() is called even if an exception is thrown inside the try block. This is a common Java 7 features interview question.

✅ Do
  • Use try-with-resources for files, DB connections, streams
  • Implement AutoCloseable in your own resource classes
  • Prefer this over try-finally for resource cleanup
❌ Don't
  • Forget to close resources (memory/connection leaks)
  • Declare resources outside the try() parentheses
  • Use try-with-resources for non-AutoCloseable objects
Section 14

Custom Exceptions

Why Create Your Own Exception?

Java's built-in exceptions are generic. In a real application — banking, e-commerce, healthcare — you need exceptions that carry your business vocabulary. InsufficientFundsException is far more meaningful in logs than a plain Exception. You can filter, search, and act on it specifically.

💡
The Two Rules for Custom Exceptions

1. Create a class that extends Exception (checked) or RuntimeException (unchecked).
2. Call super(message) in the constructor to pass the message to the parent class so getMessage() and toString() work correctly.

Creating a Custom Checked Exception

InsufficientFundsException.java
// Step 1: Create the custom exception class
public class InsufficientFundsException extends Exception {

    private double amount;  // custom field

    public InsufficientFundsException(String message, double amount) {
        super(message);  // passes message to Exception → Throwable
        this.amount = amount;
    }

    public double getAmount() { return amount; }  // custom getter
}
BankAccount.java
public class BankAccount {
    private double balance;

    public BankAccount(double balance) { this.balance = balance; }

    public void withdraw(double amount) throws InsufficientFundsException {
        if (amount > balance) {
            throw new InsufficientFundsException(
                "You don't have enough money", amount
            );
        }
        balance -= amount;
    }
}
BankTest.java
BankAccount account = new BankAccount(100.0);
try {
    account.withdraw(500.0);  // more than balance!
} catch (InsufficientFundsException e) {
    System.out.println(e.getMessage());
    System.out.println("Attempted: " + e.getAmount());
}
Output
You don't have enough money Attempted: 500.0

Checked vs. Unchecked Custom Exceptions

CustomExceptionTypes.java
// Checked custom exception — caller MUST handle or declare
public class InsufficientFundsException extends Exception {
    public InsufficientFundsException(String msg) { super(msg); }
}

// Unchecked custom exception — caller can ignore it
public class InvalidAgeException extends RuntimeException {
    public InvalidAgeException(String msg) { super(msg); }
}
📌
Why super(message) matters

The message you pass to super() travels up to Throwable's detailMessage field. This is what getMessage() returns and what toString() uses. Without calling super(message), your exception will print with no message.

⭐
Interview Tip — Best Practice in Spring/Production

In Spring Boot and production Java, custom exceptions extending RuntimeException are preferred (unchecked). This avoids polluting every method signature with throws. Common pattern: ResourceNotFoundException, ValidationException, UnauthorizedException.

✅ Do
  • Use business-meaningful names (InsufficientFundsException)
  • Always call super(message) in constructor
  • Add custom fields for more context (amount, userId)
  • Use unchecked for Spring/REST APIs
❌ Don't
  • Name your exception just MyException
  • Forget to call super(message)
  • Extend Throwable directly
  • Create exceptions for every trivial case
Section 15

Exception Chaining

The Problem: Losing the Root Cause

When you catch an exception and throw a different one, you can lose the original cause. A developer debugging the issue only sees the new exception — they have no idea what actually triggered it. Exception chaining solves this by wrapping the original exception inside the new one.

WithoutChaining.java
// ❌ Root cause lost — bad practice
try {
    connectToDatabase();
} catch (SQLException e) {
    // Original SQLException swallowed! Nobody knows what went wrong.
    throw new RuntimeException("Database connection failed");
}

The Solution: Wrap with the Cause

Pass the original exception as the second argument to the new exception's constructor. This preserves the full chain of causes so debuggers can trace the error back to its origin.

ExceptionChaining.java
public static void connectToDatabase() throws SQLException {
    throw new SQLException("Connection refused on port 5432");
}

public static void loadData() {
    try {
        connectToDatabase();
    } catch (SQLException e) {
        // ✅ Wrap original exception — root cause preserved
        throw new RuntimeException("Failed to load data", e);
        //                                                   ↑ original cause
    }
}

public static void main(String[] args) {
    try {
        loadData();
    } catch (RuntimeException e) {
        System.out.println("Message:    " + e.getMessage());
        System.out.println("Root cause: " + e.getCause().getMessage());
        e.printStackTrace();
    }
}
Output
Message: Failed to load data Root cause: Connection refused on port 5432 java.lang.RuntimeException: Failed to load data at Main.loadData(Main.java:9) at Main.main(Main.java:15) Caused by: java.sql.SQLException: Connection refused on port 5432 at Main.connectToDatabase(Main.java:2)

getCause() and initCause()

Two ways to set/get the chained cause:

CauseAPI.java
// Method 1: Pass cause in constructor (preferred)
throw new RuntimeException("High level message", originalException);

// Method 2: initCause() — set cause after construction
RuntimeException newEx = new RuntimeException("High level message");
newEx.initCause(originalException);
throw newEx;

// Retrieve the cause
Throwable cause = e.getCause();  // returns the wrapped exception
MethodDescription
getCause()Returns the wrapped (original) exception, or null if none
initCause(Throwable)Sets the cause (can only be called once)
"Caused by:" in stack tracePrinted automatically when a cause is present
⭐
Interview Tip

Exception chaining is essential for abstraction in layered architectures. A service layer shouldn't expose SQLException to the controller — it wraps it in a ServiceException. But the original SQLException should be preserved as the cause for debugging. Always wrap, never swallow.

✅ Do
  • Always pass the original exception as the cause
  • Use getCause() when debugging wrapped exceptions
  • Check for "Caused by:" in stack traces
  • Wrap low-level exceptions in high-level ones in layered apps
❌ Don't
  • Swallow exceptions silently in empty catch blocks
  • Re-throw without the original cause
  • Log and throw both (pick one or chain properly)
  • Expose low-level exceptions (SQLException) to upper layers
